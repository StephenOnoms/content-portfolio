Switching Models Without Changing Code

~6 min · Lighthouse Docs

One of the most common questions from new users is some version of: "if I want to try a different model, how much do I need to rebuild?" The short answer: nothing, beyond changing one string. This guide shows exactly what changes (and what doesn't) when you switch models, plus a few practical tips for doing it safely.

Key Takeaways

  • The model field is the only required change — request and response formats are identical across the catalog.
  • Model names follow the pattern org/model-name, matching their Hugging Face Hub identifiers.
  • Not all models support every parameter (e.g. function calling) — check a model's catalog page if you rely on advanced features.
  • Re-run your evaluation suite after switching — output quality and style can differ meaningfully between models, even with identical prompts.
  • Your billing doesn't change when you switch models on flat-rate plans, which makes this safe to experiment with.

What Changes When You Switch Models

Exactly one thing: the value of the model field in your request. Everything else — the endpoint, your authentication, the shape of messages, and the shape of the response — stays the same.

Here's the same request to three different models. Only the highlighted line changes:

# Llama 3
{
  "model": "meta-llama/Llama-3-8B-Instruct",
  "messages": [{"role": "user", "content": "Summarize this in one sentence: ..."}]
}

# Qwen 2.5
{
  "model": "Qwen/Qwen2.5-32B-Instruct",
  "messages": [{"role": "user", "content": "Summarize this in one sentence: ..."}]
}

# Mistral
{
  "model": "mistralai/Mistral-Small-24B-Instruct-2501",
  "messages": [{"role": "user", "content": "Summarize this in one sentence: ..."}]
}

Model Naming Convention

Lighthouse model names follow the organization/model-name pattern used on the Hugging Face Hub. This is deliberate — if you can find a model on the Hub, you can usually call it on Lighthouse using the same identifier (check the catalog to confirm availability). A few examples across common use cases:

A Practical Workflow for Trying a New Model

  1. Pick a small, representative sample of real inputs from your application — 10-20 examples is enough to start.
  2. Run the sample through your current model and save the outputs as a baseline.
  3. Change only the model field and re-run the same sample.
  4. Compare outputs side by side. Look for differences in tone, length, formatting, and correctness — not just "is it good," but "is it different in ways that matter for this feature."
  5. If it looks promising, expand the sample before switching in production — a 10-example check is a sanity check, not a full evaluation.

Pro tip

Because switching models doesn't change your billing on flat-rate plans, this workflow costs you nothing extra to run — which makes "just try it" a much lower-stakes proposition than on per-token pricing, where every evaluation run has a direct cost.

What Doesn't Carry Over Automatically

  • System prompt effectiveness. A system prompt tuned for one model's quirks may behave slightly differently on another — this is usually minor, but worth including in your comparison sample.
  • Advanced feature support. Function/tool calling, JSON mode, and similar features aren't supported identically across every model in the catalog. Check a model's catalog page before relying on these for a model you haven't used before.
  • Output length and verbosity. Different models have different default "chattiness." If your application parses output programmatically, check that format expectations still hold.

Common Pitfalls

  • Switching in production without re-evaluating. "Same request format" does not mean "same output quality" — always run at least a small comparison first.
  • Assuming all models support the same parameters. If a request that included a tool-calling parameter returns an error on a new model, that's the most likely cause.
  • Forgetting to update logging/labels. If you log which model handled a request (recommended — see the quickstart), remember to update this when you switch, or your analytics will misattribute results.
