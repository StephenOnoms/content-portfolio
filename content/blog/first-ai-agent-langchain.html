Building Your First AI Agent with LangChain

~14 min read · Writing sample

If you've called an LLM API and gotten a text response back, you've done the easy 90% of working with language models. The other 10% — letting the model decide what to do next, call a tool, look at the result, and decide again — is what turns an API call into an agent. This tutorial builds one from scratch using LangChain: a simple agent that can answer questions a plain LLM call can't, because it can look things up instead of guessing.

By the end, you'll have a working agent, an understanding of the loop that makes it tick, and — just as importantly — a clear picture of why agents cost more than single calls and how to keep that cost under control.

Key Takeaways

  • An agent is an LLM call wrapped in a loop that can call tools, observe results, and decide whether to call the model again or return a final answer.
  • Tools are just functions with a description the model uses to decide when to call them — the model doesn't run any code itself.
  • LangChain's role is orchestration: managing the loop, formatting tool calls and results, and tracking conversation state.
  • Agents can cost significantly more than single calls, because each loop iteration is a full model call — this is a design constraint, not an afterthought.
  • Memory turns a single agent run into a conversation, but it also means your token usage grows with conversation length unless you manage it.

What Makes Something an "Agent"

A plain LLM call is: send a prompt, get a response, done. An agent adds a loop around that: send a prompt, let the model decide whether it needs more information, and if so, call a tool to get that information, feed the result back to the model, and repeat — until the model decides it has enough to answer.

The model itself doesn't change. What changes is the scaffolding around it: a set of tools the model can choose to call, and a loop that keeps going until the model signals it's done. That scaffolding is what LangChain provides.

Prerequisites

  • Python 3.10+
  • An API key for an LLM provider with an OpenAI-compatible chat completions endpoint (most open-weight model providers offer this)
  • Basic familiarity with Python functions and virtual environments

Step 1: Project Setup

Create a virtual environment and install the packages:

python -m venv .venv
source .venv/bin/activate  # on Windows: .venv\Scripts\activate

pip install langchain langchain-openai langchain-community

Set your API key and base URL as environment variables. Using an OpenAI-compatible client means the same code works against any provider that implements that API shape — you only change the base URL and model name:

export INFERENCE_API_KEY="your-api-key-here"
export INFERENCE_BASE_URL="https://your-provider.example/v1"

Step 2: Defining a Tool

Tools are plain Python functions. The docstring matters as much as the code — it's what the model reads to decide when to call this tool. Here's a simple example: a tool that looks up a (mocked) exchange rate, something an LLM can't reliably know on its own since rates change daily.

from langchain_core.tools import tool

@tool
def get_exchange_rate(currency_pair: str) -> str:
    """Look up the current exchange rate for a currency pair,
    e.g. 'USD/EUR'. Use this whenever the user asks about
    currency conversion or exchange rates."""
    # In a real implementation, call a live exchange-rate API here.
    rates = {"USD/EUR": 0.92, "GBP/USD": 1.27, "EUR/GBP": 0.84}
    rate = rates.get(currency_pair.upper())
    if rate is None:
        return f"No rate found for {currency_pair}"
    return f"1 {currency_pair.split('/')[0]} = {rate} {currency_pair.split('/')[1]}"

Pro tip

Write tool docstrings the way you'd write documentation for a new team member who can't ask follow-up questions. Vague descriptions ("gets data") lead to the model calling the tool at the wrong times, or not calling it when it should.

Step 3: Building the Agent

With a tool defined, the agent itself is a small amount of code — LangChain handles the loop:

from langchain_openai import ChatOpenAI
from langchain.agents import create_tool_calling_agent, AgentExecutor
from langchain_core.prompts import ChatPromptTemplate
import os

llm = ChatOpenAI(
    model="your-chosen-open-weight-model",
    base_url=os.environ["INFERENCE_BASE_URL"],
    api_key=os.environ["INFERENCE_API_KEY"],
    temperature=0,
)

tools = [get_exchange_rate]

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant. Use tools when you "
                "need current information you don't already know."),
    ("human", "{input}"),
    ("placeholder", "{agent_scratchpad}"),
])

agent = create_tool_calling_agent(llm, tools, prompt)
agent_executor = AgentExecutor(agent=agent, tools=tools, verbose=True)

The temperature=0 setting isn't required, but it makes agent behaviour more consistent while you're testing — useful when you're debugging whether the model is calling tools correctly versus guessing.

Step 4: Running and Testing

response = agent_executor.invoke({
    "input": "If I have 100 US dollars, how many euros is that?"
})
print(response["output"])

With verbose=True, you'll see the agent's reasoning in the console: it recognizes it needs the current exchange rate, calls get_exchange_rate("USD/EUR"), receives the result, and uses it to compute the final answer. Try asking something the tool can't help with (e.g. "what's the capital of France?") — the agent should answer directly without calling the tool, which is exactly the decision-making this setup is meant to demonstrate.

Step 5: Adding Memory

The agent above treats each call independently — it has no memory of previous turns. To build a conversational agent, wrap it with message history:

from langchain_core.chat_history import InMemoryChatMessageHistory
from langchain_core.runnables.history import RunnableWithMessageHistory

store = {}

def get_session_history(session_id: str):
    if session_id not in store:
        store[session_id] = InMemoryChatMessageHistory()
    return store[session_id]

agent_with_memory = RunnableWithMessageHistory(
    agent_executor,
    get_session_history,
    input_messages_key="input",
    history_messages_key="chat_history",
)

response = agent_with_memory.invoke(
    {"input": "What about for 50 dollars?"},
    config={"configurable": {"session_id": "user-123"}},
)

Now the agent can resolve "what about for 50 dollars?" using the context from the previous exchange — but notice that this context is now part of every subsequent prompt, which leads directly to the next section.

Cost and Reliability Considerations for Agents

Every loop iteration in the diagram above is a full model call — which means an agent that needs three tool calls to answer a question costs roughly three to four times what a single LLM call would (the loop iterations, plus the final response). A few practical implications:

  • Set a maximum iteration limit. AgentExecutor supports a max_iterations parameter — set it explicitly. Without a limit, a model stuck in an unproductive loop (calling a tool, not liking the result, calling it again) can run far longer — and cost far more — than intended.
  • Conversation history grows every prompt's token count. With memory enabled, each new turn re-sends the accumulated history. For long conversations, this becomes the dominant cost driver — consider summarizing or truncating older history rather than sending it in full indefinitely.
  • Tool results count as tokens too. A tool that returns a large JSON blob or a long document adds all of that to the next prompt. Where possible, have tools return concise, relevant results rather than raw data dumps.
  • This is where the pricing-model choice from the rest of this series matters most. Agent workloads are inherently more variable than single-call workloads — which tends to push them toward per-token pricing, at least until usage patterns stabilize enough to evaluate flat-rate options.

Common Pitfalls

  • Vague tool descriptions. If the model calls the wrong tool, or no tool when it should, the docstring is usually the first thing to fix — before assuming the model "isn't smart enough."
  • No iteration limit. This is the single most common cause of unexpectedly high agent costs in early development.
  • Tools that fail silently. If a tool raises an exception or returns an unhelpful error message, the model may loop trying different approaches. Return clear, descriptive errors from tools, the same way you'd design error messages for a person.
  • Unbounded memory. A conversational agent that never trims history will have steadily increasing per-turn costs and, eventually, may exceed the model's context window entirely.

FAQ

Do I need LangChain specifically, or could I build this myself?

You can absolutely build the loop yourself — it's not magic, just a while loop that calls the model, checks for tool calls, executes them, and feeds results back. LangChain's value is in the parts around that loop: standardized tool definitions, prompt formatting, memory integration, and compatibility across providers, which save time especially as an agent grows more complex.

How many tools is "too many" for one agent?

There's no fixed number, but every tool description adds to the prompt the model sees on every decision, and more tools mean more opportunities for the model to pick the wrong one. If an agent's tool list is growing large, it's often a sign that it should be split into multiple specialized agents rather than one agent with many tools.

Can this agent use a different model for different steps?

Yes — and it's a common cost-optimization technique. A smaller, cheaper model can often handle the "decide which tool to call" step, while a larger model is reserved for generating the final response. LangChain supports configuring different models for different parts of a pipeline, though it adds complexity worth taking on only once cost at scale justifies it.

Final Thoughts

An agent is a small idea — let the model decide what to do next, in a loop — wrapped in just enough scaffolding to make that idea reliable: tool definitions, a loop with limits, and (optionally) memory. The code to get a basic agent running is short. The work that actually matters for production is everything in the "cost and reliability" section above — and it's worth thinking about from the first version, not retrofitting after the first surprising bill.
